# Сеть и прокси

`agsekit` поддерживает три основных сетевых helper'а:

- `proxychains` - возможность запускать агентов завёрнутыми в proxychains4
- `http_proxy` - возможность запускать агентов с предварительным запуском HTTP-proxy и заданием переменных окружения HTTP_PROXY
- `portforward` - проброс портов внутри и наружу ВМ средствами ssh-туннеля

## Proxychains

Если у вас ограниченная сеть, и есть socks5/http/https прокси-сервер который позволяет подключаться к агенту, вы можете вызывать как 
установщик агента, так и самого агента через proxychains4.

Для подробной информации "что такое proxychains" обратитесь к его документации https://github.com/haad/proxychains.

Вкратце же это возможность заставить приложение отправлять весь свой трафик через прокси-сервер, даже если приложение не поддерживает работу с proxy

Его можно задавать:

- на уровне VM в конфиге `vms.<vm_name>.proxychains: scheme://host:port`;
- на уровне агента в его конфиге `agents.<agent_name>.proxychains: scheme://host:port`
- на уровне команды через `--proxychains scheme://host:port`;
- отключать на один запуск через `--proxychains ""`

**Пример:**

1. У вас заблокирован доступ к нейросети Qwen или репозиторию NPM откуда ставится агент qwen. 
2. Однако, у вас есть возможность подключиться по SSH к некому ПК/серверу с которого такой доступ есть
3. Вы можете на вашей хостовой системе подключиться к тому удалённому серверу по SSH с аргументом -D 127.0.0.1:8080 - и вот у вас появился socks5-прокси на этом порту.
4. Дальше, вы можете прокинуть этот порт внутрь ВМ с помощью portforward (об этом дальше)
5. И включите в настройках ВМ `vms.<vm_name>.proxychains: socks5://127.0.0.1:8080` либо в настройках агента: `agents.<agent_name>.proxychains: socks5://127.0.0.1:8080`

## HTTP_PROXY

Если ваш агент не поддерживает работу через proxychains, но зато умеет работать через HTTP_PROXY, и уважает соответствующую env-переменную,
вы можете воспользоваться http_proxy. 

Таким агентом является, например, Claude-code.

Его можно задавать:

- на уровне VM в конфиге `vms.<vm_name>.http_proxy: scheme://host:port`;
- на уровне агента в его конфиге `agents.<agent_name>.http_proxy: scheme://host:port`
- на уровне команды через `--http-proxy scheme://host:port`;
- отключать на один запуск через `--http-proxy ""`

Есть разные способы задать http_proxy:

### Direct Mode

Если у вас в ВМ уже есть открытый порт на котором случае готовый к работе http-proxy, и нужно просто прокинуть эту информацию в env-переменную HTTP_PROXY, тогда в конфигурации можно задать так:

```shell
# Можно задать для ВМ
vms:
  agent-ubuntu:
    ...
    http_proxy:
      # url - это как раз адрес к уже имеющемуся http-proxy
      url: http://127.0.0.1:8080
      
# Можно задать для агента
agents:
  qwen:
    type: qwen
    http_proxy:
      # url - это как раз адрес к уже имеющемуся http-proxy
      url: http://127.0.0.1:8080
```


Вы передаёте готовый URL HTTP proxy, а `agsekit` просто выставляет `HTTP_PROXY` и `http_proxy` в окружении агента.

### Upstream Mode

Если у вас есть socks5/socks4 прокси, которые нельзя напрямую использовать как http-proxy, можно указать его,
и в этом случае запустится временный http-прокси `privoxy`

Полная запись:
```shell
# Можно задать для ВМ
vms:
  agent-ubuntu:
    ...
    http_proxy:
      # socks-proxy на этом порту
      upstream: socks5://127.0.0.1:5000
      # Слушать http-proxy на этом порту
      listen: 127.0.0.1:8080
      
# Можно задать для агента
agents:
  qwen:
    type: qwen
    http_proxy:
      # url - это как раз адрес к уже имеющемуся http-proxy
      upstream: socks5://127.0.0.1:5000
      # слушать http-proxy на этом порт
      listen: 127.0.0.1:8080
```

Если не указывать listen, для него будет выбран случайный свободный порт из диапазона `global.http_proxy_port_pool`

Если в listen указать только порт, например 8080, это будет эквивалентно "127.0.0.1:8080"

Краткая запись:

По умолчанию запись вида `http_proxy: socks5://127.0.0.1:5000` - это сокращённая форма записи:
```yaml
...
    http_proxy: 
      upstream: socks5://127.0.0.1:5000
...
```

Флаг `--http-proxy` использует именно этот upstream-режим.

## Взаимное исключение

Для одного `run` нельзя иметь одновременно `http_proxy` и `proxychains`. `agsekit` завершает запуск с ошибкой, а не пытается угадывать, какой транспорт важнее.

## Port Forwarding

Команда `agsekit portforward`, а также демон, запускаемый через `agsekit systemd start`
Создаёт и поддерживает SSH туннели внутри и наружу ВМ и периодически перечитывает конфиг, чтобы переподключать forwards при изменении правил.

Если вы не знаете что такое SSH-туннели, почитать про них можно здесь: https://docs.oracle.com/en/operating-systems/oracle-linux/openssh/openssh-SettingUpPortForwardingOverSSH.html

Вкратце же они позволяют открыть внутри ВМ порт который позволяет подключаться к сервису снаружи, а также порты снаружи, позволяющие подключиться к сервису внутри.

Пример конфигурации:
```shell
vms:
  agents-ubuntu:
    ...
    port-forwarding:
      # Типы бывают: remote, local, socks
      # remote - значит подключение изнутри ВМ на хост
      - type: remote
        host-addr: 127.0.0.1:5432 # Прокидываем хостовый PostgreSQL
        vm-addr: 127.0.0.1:5432
      # local - значит подключение из хоста в ВМ
      - type: local
        host-addr: 127.0.0.1:8080 # Подключаемся к nginx работающему внутри ВМ снаружи
        vm-addr: 127.0.0.1:80
      # socks5 - открыть в ВМ порт socks5-прокси ведущий на хост
      - type: socks5
        # Бывает полезно при сложном роутинге, когда по какой-то причине ВМ не может получить доступ
        # к адресу, к которому основная ОС прекрасно подключается
        vm-addr: 127.0.0.1:11800 
```

Конфигурацию в процессе работы можно менять, и portforward автоматически обновит туннели, закрыв не нужные и подняв новые нужные.

## Типовые сценарии

- У вас есть корпоративный SOCKS proxy для доступа к моделям
- Внутри ВМ есть веб-сервер, например nginx, и вы хотите смотреть на его выдачу в основной ВМ
- Хотите дать доступ ВМ к вашей локальной БД PostgreSQL по адресу 127.0.0.1, чтобы не колдовать с pg_hba.conf
