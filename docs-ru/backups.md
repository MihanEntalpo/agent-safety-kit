## Модель

- Снапшоты создаются на стороне хоста.
- Данные копируются через `rsync`.
- Неизменившиеся файлы hardlink'аются из предыдущего снапшота.
- Один каталог назначения защищён файловым lock, поэтому в него пишет только один backup writer 
  - Таким образом, если в одной и той же папке запущено 2 и более агента, бэкапы будут делаться только 1 раз а не в N раз чаще
- Если между последним бэкапом и текущим состоянием папки ничего не изменилось - бэкап не делается
  - В результате, агент про которого вы забыли, не создаст вам 100 ничем не отличающихся бэкапов

## Во время `agsekit run`

Если для выбранного mount включены backups и не передан `--disable-backups`:

1. `agsekit` убеждается, что initial snapshot существует, и по умолчанию также делает блокирующий pre-run snapshot, когда снапшоты уже есть;
2. агент запускается внутри VM;
3. repeated backups продолжают работать в фоне всю сессию.

Effective policy для pre-run backup определяется так:

1. `--first-backup` принудительно включает блокирующий pre-run snapshot.
2. `--no-first-backup` отключает его, если снапшоты уже существуют.
3. Если CLI override не передан, используется `mounts[].first_backup`.
4. Если backup-цепочки ещё нет, initial snapshot всё равно создаётся независимо от этого.

Если передан `--disable-backups`, фоновые repeated backups не запускаются, но initial/pre-run snapshot всё равно может быть выполнен по правилам выше.

Если `agsekit run` запущен либо из отсутствующей папки проекта, либо из существующей папки вне настроенных mount, и пользователь соглашается на временную workdir внутри VM, mount не выбирается и бэкапы для этой сессии не запускаются.

## Политики очистки

- `tail`: оставить N последних снапшотов
- `thin`: логарифмическое прореживание: плотная история рядом с текущим временем и всё более редкие старые снапшоты

## `.backupignore`

`agsekit` читает `.backupignore` внутри исходного дерева и прокидывает exclusion logic в `rsync`.

Пример .backupignore:

```text
venv/
node_modules/
*.log
!logs/important.log
```
